# AI Agent Operating Instructions

- **Language rule:** No matter what language you receive, output **English only** for code, comments, chat, documents, logs, and everything else.
- **Project scope:** Implement a **GraphQL API server** and a **remote MCP server** that provides AI-assisted features; server code is **Go**, web frontend is **TypeScript/React** under `web/`.
- **Sensitive local info location:** Local tools and debugging-related sensitive information is stored in `.github/instructions/laisky.instructions.md` (treat as sensitive; do not leak).
- **MCP multi-tenancy:** Every MCP project runs as a multi-tenant SaaS; derive `user_id` from `api_key`, strictly isolate all user data, and only allow a user to view/edit their own data.
- **JS/TS package manager:** Use **pnpm** for all package management in the `web` directory (no npm/yarn).
- **File length limits:** Every non-generated code file must be ≤800 lines; for Go, ideally keep a single file ≤600 lines; if exceeded, split by functionality/responsibility.
- **Debug logging discipline:** When debugging, add targeted **DEBUG** logs with essential details to pinpoint hard issues; after debugging, keep logs that remain useful, but **never include secrets** (API keys, passwords, tokens).
- **Concurrent agents:** Multiple agents may modify code simultaneously; preserve others’ changes, avoid interfering, and only stop and inform the user if you hit an **irreconcilable conflict**.
- **Task tracking:** Use the **TODOs tool** to track tasks and progress (keep it current).
- **Security—constant time:** Always use **constant-time comparison** for sensitive data (e.g., secrets, hashes, tokens).
- **Security—password hashing:** Follow OWASP recommendations for password hashing iterations (minimum **10,000** iterations in this context).
- **Security—SQL injection:** Never use user input directly in DB queries; always **sanitize and validate** inputs before using them in queries/conditions.
- **Time handling:** Always use **UTC** for servers, databases, and APIs (no local time).
- **Date-range semantics:** For any date-range query, the end date must include the entire final day—query must end **just before 00:00 of the next day**.
- **Testing expectations:** Add/extend **unit tests** for new features and fixes; when a new issue arises, update tests during the fix to cover it; avoid temporary one-off scripts; continuously improve the unit test suite.
- **Test assertions:** Use `github.com/stretchr/testify/require` for assertions in tests.
- **Comments requirement:** Every function/interface must have a comment explaining purpose, parameters, and return values; comment must start with the function/interface name and be complete sentences.
- **Go version/style:** Code targets **Go 1.25**; use the newest Go syntax/features where appropriate.
- **Context propagation:** Whenever feasible, use `context.Context` to manage lifecycle across the call chain (thread it through).
- **Error handling—always handle:** Handle all errors, and handle them as close to the source as possible; never ignore errors.
- **Error handling—no `err == nil`:** Never use `err == nil` (avoid patterns that risk shadowing/clarity issues).
- **Error wrapping library:** Use `github.com/Laisky/errors/v2` everywhere; never return bare errors—always wrap with `errors.Wrap` / `errors.Wrapf` / `errors.WithStack` (audit all files).
- **Error single-processing rule:** Each error must be processed exactly once—either **returned** or **logged**, but never both.
- **ORM philosophy:** ORMs can be inefficient; prefer the least DB pressure approach—use ORM mainly for **writes/updates**, and prefer **SQL for reads/conditions** whenever possible.
- **GORM read guidance:** For reads, prefer `Model/Find/First` and express conditions via SQL joins/where; use `Scan` only when the result shape doesn’t match table structure.
- **Request-scoped logging:** Any code path invoked by a request must use `gmw.GetLogger(c)` (not the global `logger.Logger`) because it carries request context.
- **Logger best practices:** Call `gmw.GetLogger(c)` only once per function and store it locally; log errors with `zap.Error(err)`; prefer structured Zap fields over `fmt.Sprintf`; never swallow errors—return or log (but not both).
- **CSS rules:** Avoid `!important`; if you think you need it, refactor CSS to remove the need; avoid inline styles in HTML/JSX—use CSS classes.
- **Web console logging:** When debugging in the web console, avoid logging objects; log **strings only** so output is easy to copy and analyze.
- **Planning discipline:** Before any action (tool call or response), independently plan: check constraints/prereqs, order operations to avoid blocking later steps, assess risks, explore likely hypotheses, adapt to new observations, and use all available info sources.
- **Grounding/precision:** When referencing rules/constraints, quote the exact applicable instruction; be precise and situation-specific, not generic.
- **Completeness:** Exhaustively incorporate all requirements/options; don’t prematurely discard low-probability causes; ask the user only when missing info is truly required for the next step.
- **Persistence:** Don’t give up early; on transient errors, retry until an explicit retry limit is reached; on non-transient errors, change strategy rather than repeating.
- **Action finality:** Only act after reasoning is complete—once you take an action, you can’t take it back.
