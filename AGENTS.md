# Agent Instructions

- **Read private instructions FIRST:** Whenever `.github/instructions/laisky.instructions.md` exists, you MUST read and follow it before any other work; it holds sensitive environment-specific details (remote dev host, API keys, private tooling, run commands) and its contents must never be leaked, quoted, logged, or committed. Also honor `.github/instructions/memory.instructions.md`.
- **Project:** `laisky-blog-graphql` (module `github.com/Laisky/laisky-blog-graphql`) is the GraphQL backend for laisky-blog, built on **gqlgen + gin**. It also ships a remote **MCP server** and a **React web frontend**. Layout: `cmd/` is the Cobra CLI (`api`, `import`, `migrate`, `tui`, plus `eval-plugin` and `promote-pageindex`); `internal/web/` is the gqlgen server whose GraphQL subgraphs (each with a `schema.graphql` registered in `gqlgen.yml`) are `blog`, `twitter`, `telegram`, `general`, and `arweave` — `blog`/`twitter`/`telegram` carry the full `controller` → `service` → `dao` → `dto`/`model` stack, while `general` and `arweave` are lighter (`arweave` is `controller` + `dto` only); `jav` is a model-only package with no GraphQL schema; `internal/mcp/` is the MCP server (`memory`, `rag`, `files`, `storage`, `tools`, `auth`, …); `internal/library/` and `library/` hold shared code (DB drivers, auth, jwt, billing, search, logging); `web/` is the React 19 + TypeScript + Vite + Tailwind frontend (pnpm). Unlike other repos, frontend work IS in scope here.
- **GraphQL codegen:** The GraphQL API is generated by gqlgen. `internal/web/generated.go` and `internal/library/models/models.go` are generated artifacts — never hand-edit them. Edit the `*.graphql` schemas listed in `gqlgen.yml` (and the resolvers/models), then regenerate with `make gen` and keep the diff in the same change.
- **Source of truth:** Keep code, comments, schema, manuals, and secondary docs aligned with `docs/` (`requirements/`, `proposals/`, `arch/`, `spec/`, `manual/`) and with shipped CLI/GraphQL/MCP/frontend behavior; if a requested change conflicts with them, resolve the conflict explicitly instead of inventing a third behavior. Some proposals are CI-enforced (see the repo-specific invariants below) — treat those as binding.
- **English only:** Output **English** for all code, comments, chat, documents, logs, and UI text, regardless of input language.
- **Secrets:** Treat any API keys or secrets in the repo or instructions as sensitive — never echo, log, commit, or expose them.
- **Research over memory:** Do NOT trust model memory for tools, platforms, APIs, or best practices — verify via web research against current (2026) industry sources before implementing; save important durable findings, categorized, under `docs/ref/` so later work can cite them instead of re-researching.
- **Testing & CI (Go):** After Go changes run `make lint` and `go test -race -cover ./...` from the repo root (CI additionally runs `-coverprofile -covermode=atomic` and `go test -bench ./...`; `make test` shells out to a legacy `tox` target — prefer the direct `go test` command). Create/update unit tests for new features and bug fixes instead of one-off scripts; continuously improve coverage; assert with `github.com/stretchr/testify/require`; never bypass a failing check — fix the test or the implementation.
- **Testing & CI (frontend):** For changes under `web/`, run `pnpm run lint` (eslint), `pnpm run test` (vitest), and `pnpm run build` (`tsc -b && vite build`) in `web/`; co-locate `*.test.tsx` tests next to the code they cover.
- **Repo-specific invariants (CI-enforced):** `make lint` runs two custom gates that also run in CI — do not break them. (1) `check_pure_go.sh`: the pageindex memory plugin under `internal/mcp/memory/plugins/pageindex/` must stay **pure Go** — no Python sidecar, cgo, `os/exec`, or IPC bring-up (per `docs/proposals/mcp_memory_plugin_manager.md` §0). (2) `check_system_owner.sh`: every SQL statement in `internal/mcp/files/` touching a tracked `mcp_file*` table must include an explicit `system_owner = ?` predicate, or carry a `// system_owner-checked: <reason>` annotation on the preceding line (§3.7).
- **Acceptance beyond existing tests:** Do NOT treat the current test suite or current code behavior as the acceptance baseline — derive the REAL intended behavior from the overall implementation and the single-source-of-truth requirement/proposal/arch docs, then build a comprehensive test set (behavior tests and an e2e matrix) to surface genuine defects; fix and re-verify them before declaring acceptance.
- **Go style:** Target **Go 1.26** with modern syntax; thread `context.Context` through call chains for lifecycle and cancellation; every exported function, type, and interface needs a doc comment starting with its name; keep lines within the `lll` cap of **256 columns**; keep files focused and cohesive rather than sprawling.
- **Errors:** Use `github.com/Laisky/errors/v2` — never return bare errors, wrap with `errors.Wrap`/`errors.Wrapf`/`errors.WithStack`; handle errors as close to the source as possible; never swallow or shadow them; process each error exactly once — either **returned** or **logged**, never both.
- **Logging:** Use structured `github.com/Laisky/zap` logging with `zap.Error(err)` (no `fmt.Sprintf` messages); on request paths pull the context-aware logger via `gmw.GetLogger(ctx)`/`gmw.GetLogger(c)` (gin-middlewares/v7) **once per function** and store it locally; the `library/log` global logger is foundation-only; add targeted DEBUG logs to aid diagnosis and keep the useful ones; **never** log secrets (API keys, passwords, tokens).
- **Security:** Use constant-time comparisons (`crypto/subtle.ConstantTimeCompare`) for tokens and signatures; hash passwords through the project `gcrypto` helpers (`go-utils/v6/crypto`) rather than ad-hoc hashing; escape user-supplied strings with `html.EscapeString` when composing HTML/GraphQL responses (past reflected-XSS bugs) and enforce length/format limits early in controllers (guards CPU-exhaustion DoS); never use untrusted input directly in DB queries or allocations — always validate and sanitize.
- **DB & time:** This service spans MongoDB, PostgreSQL (with pgvector), Redis, Firestore, and Arweave — minimize DB pressure (explicit SQL for complex reads/joins, ORM convenience for modifications; prefer Mongo `EstimatedDocumentCount` over `CountDocuments` on large collections); use **UTC** for all server, database, and API time handling; date-range queries include the entire final day (end just before 00:00 of the next day).
- **Concurrency & tracking:** Multiple agents may edit code concurrently — **preserve others' changes**, avoid overwriting, halt and notify only on an irreconcilable conflict; use the TODOs tool to track tasks and keep it current and actionable.
- **Method:** Plan before acting — check constraints and prerequisites, order operations correctly, prefer acting with available info over blocking on optional details (actions are final once executed); debug by generating and prioritizing hypotheses, testing the most likely first, and revising the plan when observations disprove one; when an anomaly is found, REPRODUCE it with a behavior test to rule out false positives BEFORE fixing, then accept the fix by turning that same test red→green; retry transient errors up to a reasonable limit and change strategy for non-transient failures; ask the user only when missing info blocks the next step.
- **Sub-agents:** Actively use sub-agents for **parallelism** (dispatch independent subtasks concurrently in a single batch) and **context isolation** (offload searches/sweeps/lookups); each must follow these same instructions, stay scoped to a well-defined subtask, and return only **distilled key information** — never raw tool output or full file dumps.
- **Code search:** Prefer `ast-grep --lang <language> -p '<pattern>'` for syntax-aware code search; use plain-text search only when explicitly requested or ast-grep does not fit.
